# Internationalization (i18n) Guide

This document explains how internationalization works in the Apex documentation site and how to add new languages.

## Overview

The documentation site uses [next-intl](https://next-intl-docs.vercel.app/) for internationalization with support for 5 languages:

- 🇺🇸 **English** (en) - Default
- 🇸🇦 **Arabic** (ar) - RTL
- 🇪🇸 **Spanish** (es)
- 🇫🇷 **French** (fr)
- 🇩🇪 **German** (de)

## Architecture

### File Structure

```
src/i18n/
├── config.ts          # Locale configuration
├── request.ts         # Server-side locale detection
├── navigation.ts      # Navigation helpers
├── utils.ts           # i18n utilities
└── messages/          # Translation files
    ├── en.json
    ├── ar.json
    ├── es.json
    ├── fr.json
    └── de.json
```

### Locale Configuration

The [`config.ts`](src/i18n/config.ts) file defines:

- **locales**: Array of supported locale codes
- **defaultLocale**: Fallback locale
- **localeLabels**: Display names, direction (LTR/RTL), and flags
- **rtlLocales**: Array of RTL locales

### Routing

The app uses Next.js App Router with dynamic `[locale]` segments:

```
app/
├── layout.tsx          # Root layout (no i18n)
└── [locale]/           # Locale-specific routes
    ├── layout.tsx      # Locale layout with providers
    └── page.tsx        # Locale-specific pages
```

## Adding a New Language

### 1. Update Configuration

Add the new locale to [`src/i18n/config.ts`](src/i18n/config.ts):

```typescript
export type Locale = "en" | "ar" | "es" | "fr" | "de" | "it"; // Add new locale

export const locales: Locale[] = ["en", "ar", "es", "fr", "de", "it"];

export const localeLabels: Record<Locale, { label: string; dir: "ltr" | "rtl"; flag: string }> = {
  // ... existing locales
  it: { label: "Italiano", dir: "ltr", flag: "🇮🇹" },
};
```

### 2. Create Translation File

Create `src/i18n/messages/it.json` by copying from `en.json`:

```bash
cp src/i18n/messages/en.json src/i18n/messages/it.json
```

### 3. Translate Content

Update all values in the new translation file:

```json
{
  "sidebar": {
    "gettingStarted": "Iniziare",
    "introduction": "Introduzione",
    // ... translate all keys
  }
}
```

### 4. Validate Translations

Run the validation script:

```bash
npm run i18n:validate
```

### 5. Sync Missing Keys (Optional)

If there are missing keys, run:

```bash
npm run i18n:sync
```

This will add missing keys with `[NEEDS TRANSLATION]` markers.

### 6. Test

Start the development server and test the new locale:

```bash
npm run dev
```

Visit `http://localhost:3000/it`

## Translation File Structure

Translation files use a nested object structure:

```json
{
  "sidebar": {
    "gettingStarted": "Getting Started",
    "introduction": "Introduction"
  },
  "home": {
    "title": "Apex Documentation",
    "description": "A utility-first CSS framework",
    "gettingStarted": {
      "title": "Getting Started",
      "description": "Learn how to..."
    }
  },
  "common": {
    "default": "Home"
  }
}
```

## Using Translations

### In Server Components

```tsx
import { useTranslations } from "next-intl";

export default function MyPage() {
  const t = useTranslations("sidebar");
  
  return <h1>{t("gettingStarted")}</h1>;
}
```

### In Client Components

```tsx
"use client";

import { useTranslations } from "next-intl";

export function MyComponent() {
  const t = useTranslations();
  
  return <button>{t("common.save")}</button>;
}
```

### With Nested Keys

```tsx
// Translation: { "home": { "title": "Welcome" } }
const t = useTranslations("home");
<h1>{t("title")}</h1>;

// Or with full path
const t = useTranslations();
<h1>{t("home.title")}</h1>;
```

## RTL Support

Arabic (ar) uses RTL direction. The framework automatically handles:

- HTML `dir` attribute
- CSS `rtl:` variants for Tailwind classes
- Logical properties (start/end instead of left/right)

Example RTL-aware styling:

```tsx
<div className="ml-4 rtl:ml-0 rtl:mr-4">
  {/* Content */}
</div>
```

## Best Practices

### 1. Use Descriptive Keys

```json
// Good
"button.submit": "Submit"
"button.cancel": "Cancel"

// Avoid
"button1": "Submit"
"button2": "Cancel"
```

### 2. Group Related Translations

```json
{
  "form": {
    "email": "Email Address",
    "password": "Password",
    "submit": "Sign In"
  }
}
```

### 3. Keep Translations Complete

Always run validation before committing:

```bash
npm run i18n:validate
```

### 4. Use Placeholders for Dynamic Content

```json
{
  "welcome": "Welcome, {name}!"
}
```

```tsx
const t = useTranslations();
<p>{t("welcome", { name: userName })}</p>
```

## Troubleshooting

### Missing Translation Error

If you see a missing translation error:

1. Check that the key exists in all translation files
2. Run `npm run i18n:validate` to identify missing keys
3. Run `npm run i18n:sync` to copy missing keys

### Locale Not Found

If a locale returns 404:

1. Verify the locale is in the `locales` array in `config.ts`
2. Ensure the translation file exists in `src/i18n/messages/`
3. Check that `generateStaticParams` includes the locale

### RTL Layout Issues

For RTL languages:

1. Use logical CSS properties where possible
2. Apply `rtl:` variants for directional styles
3. Test with browser dev tools RTL toggle

## Scripts

| Script | Description |
|--------|-------------|

| `npm run i18n:sync` | Sync missing keys from reference locale |
